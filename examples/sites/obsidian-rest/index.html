Obsidian REST API → Alap menus

Same protocol family as :obsidian:core:, but the lookups go through the Obsidian Local REST API plugin running inside the Obsidian app instead of reading Markdown off disk. The Node server holds the API key and the plugin URL — the browser just posts expression segments and gets back a list of AlapLink objects. No cert install, no key in the page, no direct browser-to-plugin call.

REST vs Core — which mode and why

Two ways to wire :obsidian: into a page. Same expression syntax, very different operational shape.

REST (this page) Core (port 9178)
Reads from The Obsidian app's Local REST API plugin. Markdown files on disk.
Setup Install the Local REST API plugin in Obsidian, configure an API key, and point the Node server at it. Node server + vault path. Obsidian doesn't need to be installed or running.
Search Obsidian's own search engine — relevance ranking, full-text, fuzzy matching, the metadata cache (backlinks, tag aliases, link graph). Substring grep over title, tags, body, path.
Best for When Obsidian is already running on the same machine and you want richer search (relevance ranking, full-text, fuzzy) or to leverage Obsidian's own indexes (backlinks, link graph, plugin-aware content like Dataview). When you don't want the search server itself to depend on Obsidian running — the markdown directory might not even be an Obsidian vault. Simple substring matches are enough.
Click-through Same in both modes: results carry obsidian://open?vault=…&file=… URIs, so clicking opens the note in Obsidian if you have it installed locally. Override linkTemplate in the protocol config to emit different URLs (e.g., a public-web preview).

The expression syntax is identical — :obsidian:rest:foo: here, :obsidian:core:foo: over there. Switching modes is changing one segment, not rewriting the page.

Clicking a result

Each link's URL is an obsidian://open?vault=…&file=… URI. With Obsidian running and the vault registered, your browser will hand the URL to it and the note will open in the app.
