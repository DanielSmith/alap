# Obsidian Vault Menus (`:obsidian:core:`)

Browser menus powered by a real Obsidian vault. The page uses the
`:obsidian:core:` protocol to turn notes into dynamic menu items. All
filesystem work happens server-side — the browser never sees the vault
path or reads from disk.

## Architecture

```
 browser                Node server                Obsidian vault
┌────────┐  POST   ┌──────────────────┐  fs.readFile  ┌─────────┐
│  main  ├────────►│ /api/obsidian/   ├──────────────►│  *.md   │
│  .mjs  │◄────────┤    query         │◄──────────────┤  notes  │
└────────┘  JSON   └──────────────────┘     text      └─────────┘
            links           ▲
                            │ obsidianHandler
                            │ (core mode)
                            ▼
                     alap/protocols/obsidian
```

Everything sensitive (the vault path) stays on the server. The browser
sends expression segments; the server resolves them against the vault
and returns a plain list of `AlapLink` objects.

## Prerequisites

1. Build the library once so `alap/protocols/obsidian` and `alap.iife.js`
   are available:

   ```bash
   cd /usr/local/projects/alap/alap
   pnpm build
   ```

2. Have an Obsidian vault on disk. The default path is
   `../obsidian-shared/AlapDocs/` — committed in this repo and
   regenerated by `pnpm -w run vault:docs` and `pnpm -w run
   vault:demo`. The directory name doubles as the Obsidian vault
   name, so opening that folder in the Obsidian app gives you a
   vault named "AlapDocs" automatically — no rename needed for the
   click-through links to work. Override via `ALAP_OBSIDIAN_VAULT`
   to point at any other vault.

## Running

```bash
./serve.sh
```

Then open http://localhost:9178/.

Environment overrides:

| Variable                    | Default                        |
|-----------------------------|--------------------------------|
| `PORT`                      | `9178`                         |
| `ALAP_OBSIDIAN_VAULT`       | `../obsidian-shared/AlapDocs/` |
| `ALAP_OBSIDIAN_VAULT_NAME`  | `AlapDocs`                     |

## What the page does

- Lists a few `.alap` triggers whose `data-alap-linkitems` attributes
  use `:obsidian:core:<query>:`.
- On load, `main.mjs` scans the page for `:obsidian:` expressions and
  pre-resolves them via the server proxy — menus open instantly once
  the page finishes loading.
- Each resolved item's `url` is an `obsidian://open?vault=…&file=…`
  URI. Clicking hands the URL to your OS; Obsidian opens the note if
  the vault name matches a configured vault.

## Files

| File          | Purpose                                                  |
|---------------|----------------------------------------------------------|
| `server.mjs`  | Node HTTP server; static files + `/api/obsidian/query`   |
| `index.html`  | Static page with Alap triggers                           |
| `main.mjs`    | Registers the client-side proxy protocol, boots AlapUI   |
| `config.mjs`  | Browser config (no secrets — vault path stays on server) |
| `styles.css`  | Minimal styling                                          |
| `serve.sh`    | `node server.mjs` with sensible defaults                 |

## Notes

- `:obsidian:core:` is **Node-only**. It's exported under the
  `alap/protocols/obsidian` subpath and is deliberately NOT part of
  `alap` or `alap/slim` — browser bundles never see it.
- The handler applies standard vault hygiene: `.obsidian/`, `.trash/`,
  and `node_modules/` folders are ignored; symlinks that escape the
  vault are refused; each note is read up to
  `OBSIDIAN_MAX_MATCH_BYTES` (256 KiB) to keep giant files bounded.
- Optional deps (`yaml`, `fast-glob`) upgrade frontmatter parsing and
  globbing when installed, with built-in fallbacks otherwise.

## Optional: switch this example to `:obsidian:rest:`

Core mode is the zero-config happy path — no plugin, no running
Obsidian, filesystem only. REST mode is for interactive workflows
where Obsidian is already open and you want the plugin's fuzzy search
instead of substring grep.

**Prerequisites for REST mode:**

1. Install the [Obsidian Local REST API](https://github.com/coddingtonbear/obsidian-local-rest-api)
   plugin (Community Plugins → Browse → "Local REST API").
2. Enable it in Obsidian Settings → Local REST API.
3. Copy the API key shown in the plugin's settings page.
4. Keep Obsidian running. The plugin serves the vault at
   `https://127.0.0.1:27124` (loopback only) while Obsidian is open.

**Switch the server's config** (`server.mjs`). Config holds data
only; the handler is passed at engine construction (3.2 pattern):

```js
// data-only protocol config
const config = {
  protocols: {
    obsidian: {
      vault: 'AlapDocs',                  // display name for obsidian:// URIs
      rest: {
        apiKey: process.env.OBSIDIAN_API_KEY,
        // host, port, scheme default to 127.0.0.1 / 27124 / https
        // rejectUnauthorized: false is honoured on loopback so the
        //   plugin's self-signed cert works out of the box
      },
    },
  },
  // ... allLinks etc.
};

// handler passed at engine construction
import { obsidianHandler } from 'alap/protocols/obsidian';
const engine = new AlapEngine(config, { handlers: { obsidian: obsidianHandler } });
```

**Swap the query prefix** in `index.html` from `:obsidian:core:` to
`:obsidian:rest:` on any trigger you want to hit the REST path.

**The library handles the self-signed cert for you** — no
`NODE_TLS_REJECT_UNAUTHORIZED=0` needed. The bypass is scoped
per-request and strictly gated to loopback hosts.

**Security audit.** REST mode opens an HTTP channel on your machine.
The plugin itself binds loopback-only, but if you wrap this example
server with a wider HTTP surface (publishing on a LAN, etc.), apply
the server-level defenses in the
[Obsidian hardening cookbook](../../../docs/cookbook/obsidian-hardening.md)
before shipping. The library-level safeguards (host allowlist, TLS
loopback-gating, path traversal guard, API key redaction in warn logs)
run unconditionally — they are the floor, not a substitute
for the server hardening.
